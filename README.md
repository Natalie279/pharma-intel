# pharma-intel

**Pharma 情报看板**

一个面向医药行业公开信息追踪的轻量级情报看板，用于集中查看大型药企的市值与财务变化、BD&并购动态、AI 合作事件，并支持生成飞书推送文案。

当前版本以前端单页应用为主，支持用户自行配置大模型 API、搜索服务商 API，并在浏览器本地完成数据缓存、结果筛选、事件诊断与推送预览。

<a name="product-intro"></a>
## 1. 产品简介

Pharma 情报看板聚焦大型跨国药企与国内头部药企的公开动态，主要包括四个模块：

1. 市值与财务
    * 展示目标药企的市值、营收、净利润 / EBITDA、核心产品、关键管线与简要财务摘要。
    * 支持国际 MNC / 国内头部分组筛选。
    * 支持市值排序与异动筛选。
    * 支持基于上次缓存数据进行市值、产品销售额、管线和风险关键词的异动检测。
2. BD&并购
    * 追踪药企近期 License-in、License-out、并购、资产收购、Option 交易、共同开发、共同商业化和战略合作事件。
    * 支持自动搜索、手动选择搜索范围、单家公司补查、粘贴原文解析。
    * 支持信源分级：T1、T1.5、T2、Unknown。
    * 支持检索诊断，帮助判断“没搜到”“搜到了但被过滤”“模型提取失败”“来源等级不足”等情况。
    * 支持对 T2 / Unknown 线索进行官方来源反查。
3. AI 合作
    * 追踪 Pharma 与 AI 公司、数据平台、算法平台、医学影像公司、临床试验 AI 公司等之间的合作。
    * 支持按合作方向、公司、证据等级、来源类型、信源等级筛选。
    * 支持搜索进度展示、检索诊断、手动粘贴解析和待核验线索反查。
4. 飞书推送
    * 从本地缓存的 BD&并购、AI 合作、市值异动中筛选高价值事件。
    * 支持周一 / 周三 / 周五 / 自定义推送节奏。
    * 支持生成飞书推送文案、手动编辑、一键复制、标记已推送。
    * 支持配置飞书机器人 Webhook 进行测试推送。

<a name="features"></a>
## 3. 核心特点

<a name="frontend"></a>
### 3.1 前端即用

当前版本为单页 HTML 应用，可直接部署到 GitHub Pages、Vercel、Netlify 等静态托管平台。

用户只需要在设置面板中配置：

* **Base URL**；
* **API Key**；
* **模型名**；
* 是否启用联网搜索；
* 搜索服务商；
* 搜索 API Key；
* **飞书 Webhook URL**。

即可开始使用。

<a name="multi-model"></a>
### 3.2 支持多模型 API

设置中内置了常用厂商预设，包括：

* OpenAI；
* Perplexity；
* DeepSeek；
* 通义千问；
* 智谱 GLM；
* Moonshot Kimi。

只要目标接口兼容 OpenAI Chat Completions 格式，理论上均可尝试接入。

<a name="search"></a>
### 3.3 搜索增强

BD&并购和 AI 合作模块支持接入搜索服务商，先进行真实网页搜索，再把搜索结果交给大模型结构化提取，以减少虚构交易、虚构链接和过时信息。

当前支持的搜索服务商包括：

* Brave Search；
* Serper.dev；
* Tavily。

<a name="source-tier"></a>
### 3.4 信源分级

项目将来源划分为：

* **T1**：公司公告、交易对手方公告、监管 / 交易所公告、Investor Relations、公司官网新闻稿；
* **T1.5**：合作方官网、官方社媒、高管 / 创始人账号、论文 / 会议、PR Newswire / Business Wire 等准官方来源；
* **T2**：媒体报道、行业网站、公众号、KOL、自媒体、普通社媒转述、搜索摘要；
* **Unknown**：来源不清楚、链接缺失、无法判断来源类型。

该设计用于帮助用户快速判断事件是否可以直接引用，还是需要进一步核验。

<a name="diagnostics"></a>
### 3.5 检索诊断

BD&并购和 AI 合作模块均支持检索诊断，包括：

* 搜索公司数量；
* query 数量；
* 原始搜索结果数量；
* 去重 URL 数量；
* 进入 LLM 的结果数量；
* 模型提取候选数量；
* 有效记录数量；
* 最终卡片数量；
* 被过滤数量；
* 被合并数量；
* 搜索失败数量；
* 各公司检索状态；
* 信源分布；
* T2 线索反查情况。

这有助于判断“结果少”到底是因为没有新��、搜索失败、模型过滤、还是信源不足。

<a name="ai-partnership"></a>
#### 4.3 AI 合作

用于查看 Pharma 与 AI 相关公司的合作事件。

主要功能：

* 清空并重新检索；
* 选择搜索范围；

<a name="user-guide"></a>
## 5. 用户指引

<a name="first-use"></a>
### 5.1 首次使用

1. 打开网页。
2. 点击右上角 **设置**。
3. 选择常用厂商预设，或手动填写：
    * **Base URL**；
    * **API Key**；
    * **模型名**。
4. 如需真实搜索增强，请填写：
    * 搜索服务商；
    * 搜索 API Key。
5. 如需测试飞书推送，请填写：
    * **飞书自定义机器人 Webhook URL**；
    * 是否启用 Webhook 推送。
6. 点击保存。

**注意**：当前版本的 API Key、搜索 API Key 和飞书 Webhook 配置保存在本地浏览器中，仅适合内部测试。正式团队使用建议通过后端托管密钥。

---

### 5.2 刷新市值与财务数据

1. 进入 市值与财务 Tab。
2. 点击 **刷新数据**。
3. 等待卡片生成。
4. 可使用顶部筛选项查看：
    * 全部公司；
    * 国际 MNC；
    * 国内头部公司。
5. 可按市值从高到低或从低到高排序。
6. 若系统检测到明显变化，可点击 **异动筛选** 查看命中公司。

说明：当前市值与财务数据由 AI 基于公开信息生成，可能存在延迟、近似或误差，仅供内部参考。正式引用请以公司财报、公告、交易所披露或权威数据为准。

---

### 5.3 刷新 BD&并购事件

1. 进入 BD&并购 Tab。
2. 选择搜索范围：
    * 自动；
    * 过去 7 天；
    * 过去 14 天；
    * 过去 30 天；
    * 过去 90 天。
3. 选择公司范围：
    * 全部公司；
    * 或单家公司。
4. 点击 **刷新交易**。
5. 等待搜索进度完成。
6. 使用筛选器查看特定公司、交易类型、证据等级、来源类型或信源等级。
7. 如发现 T2 / Unknown 线索，可点击 **反查官方来源**。
8. 如需查看整体检索情况，可点击 **查看检索诊断**。

建议：正式引用前，优先使用 **T1 或 T1.5** 来源。T2 来源适合作为线索，不建议直接作为正式证据。

---

### 5.4 粘贴原文解析 BD 事件

适用于微信公众号、行业文章、公司新闻稿、PDF 摘录等无法自动抓取的内容。

1. 进入 BD&并购 Tab。
2. 点击 **粘贴原文解析**。
3. 粘贴文章正文。
4. 点击 **解析**。
5. 系统会尝试从原文中提取 BD 交易信息。
6. 如果该交易已存在，系统会尝试合并；如果不存在，会新增为手动导入事件。

**注意**：手动导入事件需要用户自行核验原文出处。

---

### 5.5 刷新 AI 合作事件

1. 进入 AI 合作 Tab。
2. 选择搜索范围。
3. 选择公司范围。
4. 点击 **刷新 AI 合作**。
5. 等待搜索进度完成。
6. 使用筛选器查看：
    * 公司；
    * 合作方向；
    * 证据等级；
    * 来源类型；
    * 信源等级。
7. 如需排查结果质量，可点击 **查看检索诊断**。
8. 如需进一步分析某条合作，可点击对应卡片的 **深挖分析**。

---

### 5.6 生成飞书推送

1. 进入 飞书推送 Tab。
2. 选择推送周期：
    * 自动识别今日；
    * 周一更新；
    * 周三更新；
    * 周五更新；
    * 自定义。
3. 选择覆盖时间：
    * 自动；
    * 过去 2 天；
    * 过去 3 天；
    * 过去 7 天；
    * 全部缓存。
4. 选择最大推送条数。
5. 勾选推送内容：
    * BD&并购；
    * AI 合作；
    * 公司异动。
6. 点击 **重新生成候选与文案**。
7. 在文案预览区手动调整内容。
8. 可选择：
    * 一键复制；
    * 标记为已推送；
    * 推送到飞书 Webhook。

注意：当前推送候选只来自浏览器本地缓存。如果 BD、AI 或市值板块还没有刷新过，飞书推送中也不会有对应内容。

<a name="data-cache"></a>
## 6. 数据与缓存说明

当前版本主要使用浏览器 localStorage 存储：

* 市值与财务缓存；
* BD&并购缓存；
* AI 合作缓存；
* 深挖分析缓存；
* ��送历史；
* 用户设置；
* 飞书 Webhook 设置；
* 异动检测规则。

这意味着：

* 换设备后数据不会同步；
* 清除浏览器缓存后数据会丢失；
* 不同用户看到的数据可能不同；
* 不适合正式团队协作；
* 不适合长期历史趋势分析。

如需正式团队使用，建议后续接入后端数据库。

<a name="limits"></a>
## 7. 当前限制

### 7.1 数据准确性限制

市值、营收、利润、产品销售额等数据可能来自模型生成或搜索摘要，存在延迟、估算或误差。正式引用前应以公司公告、财报、交易所披露和权威数据为准。

### 7.2 CORS 限制

由于当前版本主要在浏览器前端运行，部分网站、搜索 API、财报 PDF、公司官网和社媒内容可能无法直接抓取。这会导致系统只能读取搜索摘要，而不是全部原始页面内容。

### 7.3 API Key 暴露风险

**重要**：当前 API Key、搜索 API Key 和飞书 Webhook URL 存储在本地浏览器中。虽然适合个人或内部测试，但不适合正式生产环境。

### 7.4 无后端数据库

当前项目无法天然支持：

* 多人共享数据；
* 团队统一推送记录；
* 跨设备同步；
* 历史趋势沉淀；
* 权限管理；
* 自动定时任务。

## 7.5 自动推送能力有限

前端页面只有在用户打开网页并手动操作时才能稳定运行。若需要周一 / 周三 / 周五自动推送，应接入后端或 Serverless 定时任务。

## 8. 推荐后续升级方向

### 8.1 轻量后端 / Serverless

优先将以下能力后端化：

* 财务快照；
* 产品销售额；
* BD 事件；
* AI 合作事件；

<a name="scheduling"></a>
### 8.3 定时任务

建议后续支持：

* 每日更新市值；
* 每周或每季度更新财务与产品销售额；
* 周一 / 周三 / 周五自动生成推送候选；
* 自动推送到飞书群；
* 推送失败重试；
* 异常告警。

<a name="market-enhance"></a>
### 8.4 市值板块增强

建议将市值板块从 AI 摘要升级为“结构化市场数据 + 财报数据 + 事件解释”模式：

* 当前市值；
* 1 日 / 7 日 / 30 日变化；
* 最近财报期；
* 收入与利润；
* Top 产品销售额；
* 核心管线；
* 市值变化原因；
* 与 BD、AI、临床、监管、财报事件的关联分析。

<a name="dev-roadmap"></a>
## 13. 开发者说明

本项目目前仍处于持续迭代阶段。当前版本已经完成了基础看板、BD&并购检索、AI合作检索、市值与财务展示、飞书推送文案生成等核心功能，但在交互与稳定性上仍有改进空间。

以下为当前开发规划与待办事项。

---

### 13.1 近期优化事项

#### 13.1.1 调整“清空并重新检索”逻辑

当前 BD&并购和 AI合作板块中的 “清空并重新检索” 按钮会在清空缓存后立即触发新一轮检索。

后续建议将其改为单纯的 **“清空”** 操作：

* 点击后只清除当前板块缓存；
* 不自动开始新一轮检索；
* 用户可自行点击“刷新交易”或“刷新 AI 合作”重新搜索；
* 避免用户误操作后立刻触发大量 API 请求；
* 降低搜索 API 和大模型 API 的消耗。

建议按钮文案调整为：

**🗑 清空缓存**

---

#### 13.1.2 未配置 API 时给出友好提示

当前如果用户尚未配置 API Key，直接点击刷新，可能会连续触发多个 401 error，体验较差。

后续应在所有需要调用大模型或搜索 API 的操作前统一增加配置检查。

建议逻辑：

如果未填写 API Key：
  不发起请求；
  不进入批量刷新流程；
  不显示多条 401 错误；
  只展示一条明确提示。

建议提示文案：

**请先在右上角「设置」中配置大模型 API Key，再刷新数据。**

如果是搜索增强功能未配置搜索 API Key，则提示：

**当前未配置搜索 API Key，将仅依赖模型自身能力，结果准确性可能下降。**

如果某个功能必须依赖搜索 API Key，例如官方来源反查，则提示：

**反查官方来源需要先在「设置」中配置搜索服务商 API Key。**

---
#### 13.1.3 BD 板块空结果状态优化

当前 BD&并购板块在结果为空时，部分情况下会显示：‘没有符合筛选条件的交易’；
但如果用户还没有刷新过，或清空缓存后没有数据，这个提示容易让用户误以为“系统已经搜索过但没有结果”。

后续建议区分两种状态：

1. 无缓存 / 未刷新状态
2. 有缓存，但被筛选条件过滤为空
---
#### 13.1.4 筛选逻辑优化：从“筛选”改为“排序 / 分组”

当前 BD&并购和 AI合作板块中存在以下筛选项：

* 全部证据等级；
* 全部来源类型；
* 全部信源等级。

这些筛选项虽然信息完整，但用户实际使用时可能意义不大。多数情况下，用户并不想“只看 T1”或“只看低证据事件”，而是希望结果按照可信度和重要性自然排序。

---
#### 13.1.5 分析卡片排版优化

当前深挖分析内容较多，阅读时容易显得密集。

后续建议优化：

* 子标题加粗；
* 分段之间增加间距；
* 关键字段使用标签式展示；
* “事实信息”和“分析判断”视觉区分；
* 缺失信息、风险、来源链接单独成块；
* 金额、权益范围、研发阶段等核心字段靠前展示；
* 长文本避免堆在同一段中。
---
#### 13.1.6 卡片字体与可读性优化

后续建议整体优化卡片层级：

* 公司名、事件标题使用更明显字号；
* 日期、来源、证据等级使用较弱视觉层级；
* 关键金额、标的、权益范围使用更醒目的展示；
* 来源链接避免过长破坏排版；
* 多标签时自动换行；
* 卡片内行距适当增加；
* 深色模式下保持足够对比度。

---
#### 13.1.7 财务板块优化

市值与财务板块目前仍主要依赖大模型生成摘要，准确性和可解释性弱于 BD&并购、AI合作板块。

建议新增能力：

* 市值搜索 / 抓取进度；
* 数据来源展示；
* 数据基准日展示；
* 财报期展示；
* 市值异动原因解释；
* 近 7 日 / 30 日变化；
* 与 BD / AI / 临床 / 监管事件的关联解释；
* 财务数据缺失原因说明；
* 财务板块检索诊断。

中长期建议将市值与财务数据迁移至后端，通过金融数据 API、财报数据 API 或官方财报解析获得，减少对模型估算的依赖。

### 13.2 需要持续迭代的配置

以下内容不应视为一次性固定配置，而应作为产品持续迭代的一部分。

#### 13.2.1 公司名单

当前公司名单覆盖部分国际 MNC 和国内头部药企，但后续需要根据实际使用场景持续调整。

需要持续维护：

* 是否新增更多国际 Pharma；
* 是否新增 Biotech；
* 是否新增中国创新药公司；
* 是否区分 MNC、国内药企、AI biotech、平台公司；
* 是否为每家公司补充 ticker；
* 是否为每家公司补充交易所；
* 是否补充官网域名；
* 是否补充 IR 页面；
* 是否补充常见别名、中英文名、历史名称；
* 是否补充子公司 / 关联公司名称。

建议公司配置逐步从前端 JS 中迁移到独立配置文件或后端数据库中，方便维护和多人协作。

⸻

#### 13.2.2 纳排标准

BD&并购和 AI合作板块的结果质量高度依赖纳入 / 排除标准。

后续需要持续校准：

BD&并购纳入标准

建议纳入：

* License-in；
* License-out；
* 并购；
* 资产收购；
* Option-to-license；
* 共同开发；
* 共同商业化；
* 有明确资产、权益范围或合作对象的战略合作。

建议排除：

* 普通供应协议；
* 纯 CDMO 服务；
* 普通销售代理；
* 无明确资产或权益范围的泛战略合作；
* 单纯融资；
* 普通产品上市；
* 临床试验启动；
* 获批新闻；
* 市场传闻；
* 无真实来源链接的事件。

AI合作纳入标准

建议纳入：

* 药物发现 AI；
* 临床试验 AI；
* 医学影像 AI；
* 真实世界数据 / RWD / RWE 合作；
* 生成式 AI 在药企内部或业务场景的合作；
* AI 模型 / 平台授权；
* 数据基础设施；
* 商业化 / 患者服务 AI；
* 明确由 Pharma 与 AI / 数据 / 算法 / 数字医疗公司共同披露的合作。

建议排除：

* 单纯产品采购；
* 普通 IT 服务；
* 纯营销活动；
* 无明确 AI 技术或业务场景的合作；
* 媒体对 AI 战略的泛泛报道；
* 没有合作方或没有具体合作内容的新闻。

纳排标准需要通过真实结果不断修正，否则会出现结果过宽或过窄的问题。

⸻

#### 13.2.3 推送标准

飞书推送的目标不是“把所有结果都推送出去”，而是帮助用户快速看到最重要、最值得关注的事件。

后续需要持续优化推送标准。

建议优先推送：

* T1 / T1.5 来源确认的事件；
* 明确金额披露的 BD 交易；
* 并购、重大 License、资产收购；
* 涉及核心治疗领域或热门机制的合作；
* 涉及头部 Pharma 与重要 AI / biotech 公司的合作；
* 对公司战略有明显影响的事件；
* 市值出现显著变化且有明确原因的事件；
* 与财报、核心产品、临床读出、监管节点相关的事件。

建议降低优先级：

* 仅 T2 来源支持的事件；
* 无金额、无资产、无权益范围的泛合作；
* 单纯媒体评论；
* 旧闻转载；
* 重复事件；
* 信息不完整且无法核验的线索。

当前版本主要是前端单页应用。为了支持更稳定的数据质量、更安全的密钥管理和团队协作，后续建议逐步引入后端能力
